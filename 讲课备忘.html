AI 与设计整合

从行业变革到工程实践

核心命题

AI 设计自动化不是工具问题。语料库的结构化程度，决定了 AI 对你设计判断的理解上限。

01 职能分工正在瓦解：通用 Builder 的兴起
LinkedIn 2025 年底正式执行
终止了运行多年的 APM（Associate Product Manager）项目，从 2026 年 1 月起替换为 Associate Product Builder 项目。新项目的目标：每个新人同时学会写代码、做设计、做产品管理，能从想法独立做到上线。

首席产品官 Tomer Cohen 原话：「我们要教他们在 LinkedIn 怎么写代码、做设计、做 PM。」Satya Nadella 也公开提到：LinkedIn 把 PM、设计师、前端、后端四个角色合并扩展，统称 Full-Stack Builder。
Meta Reality Labs 2026 年 3 月备忘录落地
通过内部备忘录正式宣布：约 1000 人的团队里，所有传统职位头衔（工程师、设计师、PM）统一替换为三个新头衔：AI Builder / AI Pod Lead / AI Org Lead。

团队按「Pod」重组，每个 Pod 刻意设计为跨职能——工程师承担设计任务是预期行为，不是例外。Zuckerberg 原话：「以前需要大团队的项目，现在可能一个非常有才华的人就能完成。」

传统团队是流水线——PM 写需求、设计师出稿、工程师实现。AI 在压低每个环节的执行成本，当「出一版稿」从两天变成两小时，维持分工的经济逻辑就动摇了。

Builder 角色的核心护城河不是「会用更多工具」，是把设计判断力结构化的能力。语料库搭建正是这件事的载体。

02 为什么语料库是核心问题

AI 工具本质上在做一件事：在你的提示和它已知的世界之间取交集。如果它已知的世界里没有你的设计标准，它就用训练数据里的「行业平均」来填空——看起来可以，但没有你的东西。

这不是工具不够好。Figma MCP、Claude Design、Figma Make 都很强——但它们的输出上限，是你给进去的设计系统的质量上限。Atlassian 的设计系统团队在 2025 年底做了一次测试：把组件文档转成结构化 JSON 喂给 AI，一次尝试里达到了约 70% 的准确度；没有这份语料的情况下，AI 会直接引用不存在的 API 名称和组件名。

你给新设计师入职看的那些东西——品牌手册、组件规范、历史设计稿——就是语料库的原料。它们现在是人类可读的，需要变成机器也可读的。

03 核心概念与原理
原始 token vs 语义 token Primitive vs Semantic Token
这是语料库里最容易踩的坑。原始 token 描述「它是什么」，语义 token 描述「它用在哪」。AI 只理解语义 token，但大多数设计师的 Figma Variables 停留在原始层。「red-6」是原始 token。「color-feedback-error」是语义 token。
上下文 vs 规则 Context vs Rule
给 AI 上下文（我的设计系统是这样的），和给 AI 规则（你必须这样做），效果完全不同。语料库的核心是上下文，不是规则。规则是在上下文不足时的补丁——上下文够好，AI 会自己推断出正确的规则。
歧义消除 Ambiguity Reduction
语料库精度的本质是消除歧义。「现代感」对 AI 来说有一百种解读；「圆角 8px，字重 500，#1D4ED8 背景」没有歧义。你每消除一个歧义点，AI 的输出就更可控一分。这不是在限制 AI，是在帮 AI 理解你的标准。
元数据层 Metadata Layer
组件不只有视觉规格，还有使用意图、禁用场景、配对关系。这些是元数据。没有元数据，AI 知道按钮长什么样，但不知道什么时候该用 primary，什么时候用 ghost。这是为什么有了 Figma Variables 还不够——Variables 是视觉值，元数据是判断逻辑。
语料库 vs 提示词 Corpus vs Prompt
改提示词是每次任务的一次性投入；建语料库是一次性投入，之后每次任务都受益。依赖提示词的人，在重复支付同一笔成本；建了语料库的人，成本在摊薄。这也是为什么语料库的搭建质量决定了自动化的天花板。
04 实践指南：要做什么，不要做什么
×
用视觉截图替代文字描述 给 AI 看一张 Figma 截图，让它「照着做」。AI 能识别截图，但识别精度远低于结构化文字。尤其是 Figma MCP 在没有 Code Connect 授权时，会静默退化成只看图片——输出大幅下降而你不知道为什么。
✓
把组件文档翻译成结构化 JSON 或 Markdown 不要让 AI 读 PDF 或 Figma 截图。把按钮的状态、色值、尺寸、用法写成结构化格式。Atlassian 的方法：把顶部导航的所有变体转成 JSON，2000 行，AI 一次尝试 70% 准确率。没有这份 JSON，AI 引用的是幻觉组件名。
×
只给 primitive token，不给 semantic token 这是最常见的坑。你的 Figma Variables 里有 red-6、blue-3、space-4，给进去之后 AI 会选错——因为它不知道 red-6 是「只用来构建语义 token 的原料，不直接用于组件」。
✓
语义 token 要附带使用说明和禁用说明 好的语义 token 告诉 AI：什么场景用这个 token，什么场景不该用，配对用哪些 token。「color-feedback-error 用于错误状态；不要用于警告（用 color-feedback-warning）；不要在深色背景上直接用作文字色」——这才是 AI 能理解的语料。
× AI 读不懂
{
  "red": {
    "6": {
      "value": "#DE3E25",
      "type": "color"
    }
  }
}
✓ AI 能理解
{
  "color-feedback-error": {
    "value": "#DE3E25",
    "description": "错误状态色",
    "usage": ["错误提示", "危险按钮"],
    "doNot": ["用于警告", "深色背景文字"],
    "components": ["Button","Alert","Input"]
  }
}
×
把所有历史设计稿全部喂进去 「给 AI 越多材料越好」是误区。没有结构的材料堆，AI 找不到主线，会把旧版设计、废弃组件、临时方案都当成有效参考。结果是输出随机。
✓
建语料后立刻测试，找到「歧义点」再迭代 写完一段语料，马上用 AI 跑一个真实任务，看输出哪里不对。不对的地方就是你的语料还有歧义的地方。改语料是一劳永逸的，改提示词是每次任务都要重做的。
×
只写视觉规格，不写判断逻辑 「主按钮：蓝色，圆角 8px，高 40px」——这是视觉规格。但 AI 仍然不知道什么时候用 primary，什么时候用 secondary，什么时候用 ghost。缺了判断逻辑，AI 会凭感觉猜，每次结果不一样。
✓
把设计决策的「为什么」写进语料 「主按钮用在最重要的单一操作上；一个视图里不应出现超过一个主按钮；在信息层级低的场景（如列表行内操作）改用 ghost」——这是判断逻辑，是语料里最有价值、也最难写的部分。
05 技术选型与建议

市面上工具很多，但选型的核心判断标准只有一个：这个工具能不能消费你的结构化语料？如果不能，它再强也无法把你的设计标准带进输出。

工具 定位与选型建议
Figma + MCP成熟 目前最完整的设计语料输入管道。设计稿通过 MCP Server 变成结构化数据，Claude Code / Cursor 等工具直接读取。 关键前提：你的 Figma 文件里 token、组件命名、规则必须是结构化的。乱的 Figma 文件通过 MCP 生成的代码也是乱的。Code Connect 功能需要 Organization 及以上授权，没有授权会静默退化成只看图片。
Claude Design2026.04 Anthropic 刚发布。Onboarding 时读你的代码库和设计文件，自动构建设计系统，后续任务都基于此生成。核心是「你给进去的设计系统有多好」。 适合：营销落地页、幻灯片、品牌创意。不适合替代 Figma 做精细组件系统管理。Credit 消耗快，重度使用费用较高。
Google Stitch2026.03 支持 DESIGN.md 标准输入，可以同时生成 5 个关联界面，直接导出到 Figma。 DESIGN.md 是 Google Labs 开源的设计系统描述格式，正在成为行业标准。趋势是：以后语料库会有统一格式，就像代码有 README 一样。
Tokens Studio成熟 管理语义 token 的最成熟工具，可以直接导出 AI-readable 的 JSON 格式，也是搭建两层 token 体系（primitive → semantic）的标准路径。 如果只用 Figma Variables 而不用 Tokens Studio，基本上停留在原始 token 层，喂给 AI 效果有限。
Supernova / Zeroheight 设计系统文档平台，可以通过 API 将 token 和组件文档暴露给 AI 消费。适合团队规模较大、设计系统已经比较完整的情况。 核心价值：让你的设计系统文档从「人类读的静态页面」变成「机器可查询的结构化接口」。

不要先选工具再想「我的语料库怎么配合这个工具」。先把语料库做好，再选能消费它的工具。

06 新模型速览：跟上一版最大的差异

重点不是参数，是每个跟上一版相比实质变了什么。

GPT-5.5 vs GPT-5.4 · 2026.04.23
最大差异：从「需要人推着走」变成「自己跑完」。GPT-5.4 在多步骤任务中有个已知的失败模式——到第 8、9 步时会忘掉最初的约束条件，遇到模糊情况就停下来问人，长任务变成半手动流程。5.5 直接解决了这个问题：遇到歧义时自己判断最可能的意图并继续，不再卡住。

另一个实质变化：长上下文从「名义上支持」变成「真正可用」。5.4 的 1M token 窗口在超长文档下准确率只有 36.6%，5.5 达到 74%——这个差距在真实任务里非常明显。速度没有变慢，token 消耗反而更少。 对设计关联：设计系统更新、跨文件组件检查、从稿子到代码到文档的一气贯通——这类需要「不被打断地跑完」的任务，5.5 比 5.4 可靠得多。
Claude Opus 4.7 vs Opus 4.6 · 2026.04.16
最大差异：视觉能力质变 + 跨会话记忆。4.6 在读高清图、密集截图、多图关联时经常出错；4.7 的视觉输入上限从约 800px 提升到 2576px（是之前的 3 倍以上），图表解读、UI 截图理解、OCR 类任务的准确率都有明显提升。

另一个实质变化：4.6 每次会话结束记忆就清空；4.7 引入了文件系统记忆，可以跨多次会话记住上次做了什么，下次任务不需要重新铺垫上下文。

有一处退步值得注意：4.7 在长文档深处提取信息的准确率低于 4.6，信息埋得越深越不可靠。 对设计关联：能读懂高分辨率设计稿和复杂图表；跨会话记忆让持续迭代语料库成为可能；也是 Claude Design 的底层模型。
Kimi K2.6 vs K2.5 · 2026.04.20
最大差异：长时间运行的稳定性，不是上下文窗口变大。最常见的误读是「K2.6 扩大了上下文」——实际上两个版本都是 256K，完全一样。K2.6 真正改进的是：长任务运行过程中的指令遵从度、自我纠错能力、复杂工程任务的稳定性。K2.5 在跑长会话时会「漂移」——越跑越偏；K2.6 解决的就是这个问题。

新增原生视频输入（K2.5 只支持图片）。开源权重，MIT 许可，价格约为 GPT-5.5 的 1/5，可私有部署。全球综合排名第 4，Agentic 基准分比 K2.5 高 18.5 分。 对设计关联：K2.5 文档侧重前端设计质量，K2.6 文档侧重长时间工程执行——适合需要持续跑设计系统维护、多品牌 token 同步等不能中途漂移的场景。对数据安全有要求的团队可私有部署。
GPT Image 2 vs DALL-E 3 · 2026.04.21
最大差异：文字渲染从「基本不可用」到「约 99% 准确」。DALL-E 3 生成含文字的图片几乎不可用——「50% OFF」经常变成「5O% 0FF」，中文、韩文、阿拉伯文更是烂掉；用户的标准做法是先生成无字图再在 Photoshop 里手动排文字。GPT Image 2 原生支持多语言文字渲染，包括中日韩文和阿拉伯文，且是先推理布局再生成，不是盲猜。

其他实质改变：分辨率从 1792×1024 升到 4K（3840×2160，原生生成不是放大）；支持最多 10 张参考图一次输入，可以做风格迁移和角色一致性；编辑时只改指定区域，不会重画整张图（DALL-E 3 编辑经常全图重来）。 对设计关联：质变，不是量变。含文字的 UI mockup、信息图、海报、品牌物料，第一次可以用 AI 直接生成可用稿，不需要后期修字。
DeepSeek V4 vs V3 · 2026.04.24
最大差异：上下文从 128K 扩到 1M，编程能力跃升 13 个百分点。V3 的 128K 上下文窗口是最大短板，限制了它处理大型代码库和长文档的能力；V4-Pro 直接扩到 1M，同时最大输出从 8K 升到 384K——这意味着可以一次性处理整个设计系统的代码库。SWE-bench（真实 GitHub 问题解决率）从 67.8% 升到约 81%，是这次最显眼的跳跃。

发布两个版本：V4-Flash（轻量，$0.14/M token，适合高频调用）和 V4-Pro（旗舰，1.6T 参数，$1.74/M token）。全程 MIT 开源，支持华为芯片，针对中国企业私有部署做了专项优化。V3 的 deepseek-chat API 将于 2026 年 7 月 24 日退役，目前已路由到 V4-Flash。 对设计关联：开源 + 超低价格 + 中文能力强 + 可私有部署——对国内团队来说是成本最低的大上下文选项。V4-Pro 的 1M 上下文可以一次读完整个设计系统文档库。

横向一句话：GPT-5.5 / Opus 4.7 在打「跑完复杂任务」的稳定性；Kimi K2.6 / DeepSeek V4 在打「开源低价可私有部署」；GPT Image 2 是图像生成的独立赛道，解决了一个卡了三年的老问题。

07 用 AI 做多 Agent 写作与任务派发

单 Agent 顺序执行是瓶颈。多 Agent 的核心价值是并行——让不同 Agent 同时处理独立任务，最后由一个主 Agent 汇总。以下是四种用法，按「人的介入程度」从高到低排列。

多窗口 + Codex App 派发 人做调度者
最直觉的方式——你自己就是 Orchestrator。同时开多个 Agent 窗口，每个负责一块任务，你在中间传递信息、审查结果、决定下一步。

最简单的版本：开几个浏览器窗口，一个写大纲、一个扩写内容、一个做格式校对，你来回切换和汇总。不需要任何工具，现在就能用。

升级版是 OpenAI 的 Codex App（2026 年 2 月发布）：专门为这种场景设计的「多 Agent 指挥中心」，每个 Agent 在独立的代码分支里工作，互不干扰，你可以检查每个 Agent 的改动、插入指令，最后统一 merge。 适合：每个节点都需要你判断和介入、任务之间逻辑上有强依赖的场景。
Worktree 并行 各自在独立分支工作
想象你有三个设计师，每人负责一个功能模块，各自在自己的草稿本上画，画完再汇总——Worktree 就是这个思路。

技术上是给每个 Agent 开一个独立的 Git 分支（worktree），Agent 在自己的分支里随便改，不会影响别人的工作。多个 Agent 可以同时对同一个代码库动手，却完全不会产生冲突，最后你逐一审查每个分支的结果，满意的 merge 进去。

跟「多窗口」的区别：多窗口是你在手动传话和协调；Worktree 并行是 Agent 各自独立跑完，你最后统一验收，中间不需要干预。 适合：同时推进多个独立功能模块，或者想对同一个问题跑多个 Agent 方案、最后挑最好的那个。
Sub-Agent（子 Agent） AI 自动派发
主 Agent 将一个大任务分解，派发给多个子 Agent 并行执行，每个子 Agent 有独立的上下文窗口，完成后把结果汇报给主 Agent 合并。子 Agent 之间不互相通信，主 Agent 也看不到它们的过程，只收最终结果。

典型用法：「用 4 个并行子 Agent 分别研究竞品 A / B / C / D，完成后汇总成对比矩阵」；或把一个设计系统分成前端组件层、token 层、文档层、测试层，四个子 Agent 同时跑。

关键限制：子 Agent 不能中途被干预，也不互相知道对方在做什么——任务必须真正独立，有依赖关系的不适合并行。
Claude Agent Teams 2026.02 发布，实验性功能
Sub-Agent 的升级版。架构上有一个 Team Lead（主协调者）和多个 Teammate（执行者），它们之间有共享任务列表和消息系统——Teammate 可以实时看到其他人在做什么，可以互相发消息，Team Lead 可以随时调整任务分配。

跟 Sub-Agent 的本质区别：Sub-Agent 像派了几个实习生分头去办事，结果才汇报；Agent Teams 像一个有站会的小团队，成员之间持续同步。当后端 Agent 改了数据结构，前端 Agent 能立刻知道，不用等到集成时才发现冲突。

启用方式：Claude Code 里按 Ctrl+T，需要在 settings.json 里开启实验性功能。配合 tmux 可以让每个 Agent 有独立的终端面板，可以实时观察和干预。 适合场景：多模块并行开发、长时间设计系统维护、需要 Agent 间动态协调的复杂任务。
多窗口 + Codex App 派发 最灵活，适合手动调度
不依赖自动编排，由人来做调度者。OpenAI 在 2026 年 2 月发布了 Codex 桌面 App，定位是「多 Agent 指挥中心」——可以同时开多个 Agent 线程，每个线程在独立的 Git worktree 里工作（互不干扰），人可以在不同线程之间切换、检查 diff、插入指令。

跟前两种的区别：前两种是 AI 编排 AI，这种是人编排 AI。适合你需要对每个子任务有高度控制的场景——比如一个 Agent 负责写内容大纲，你审完再派另一个 Agent 扩写，再派第三个 Agent 做格式校对。

多窗口的最简版本：不用任何专门工具，直接开多个 Claude 或 ChatGPT 浏览器窗口，各自处理不同模块，自己做汇总。低成本，无需配置，适合一次性任务。 Codex App 的内置 worktree 隔离是关键优势——多个 Agent 改同一个代码库不会冲突，各自在独立分支上，最后 merge。

Worktree 并行 中（最后验收） 无（各自独立分支） 多模块同时跑，结果独立审查
Sub-Agent 低（只看结果） 无（单向汇报） 完全独立的并行任务
Agent Teams 最低（AI 自动协调） 有（共享任务列表 + 消息） 模块间有动态依赖的协作

判断用哪种：你需要全程把控 → 多窗口；你想并行跑多个方案再挑 → Worktree；任务完全独立只看结果 → Sub-Agent；任务之间有动态依赖需要 Agent 互相同步 → Agent Teams。

08 行业方向（快速过）
语料库格式标准化DESIGN.md
Google Labs 2026 年 4 月开源。设计系统有了标准描述格式，就像代码有 README。这意味着未来「如何写语料」会有业界通行标准，现在先想清楚结构，迁移成本低。
设计与代码双向同步Design ↔ Code
以前是「设计稿 → 代码」单向。现在 Figma MCP 支持代码改完直接反写回 Figma，保持同步。这要求你的语料库本身是代码库和设计稿的共同锚点——两边都在读它。
设计判断力外显化Design Rationale
行业越来越清楚：设计师的核心价值不是会用工具，是有判断力。而语料库搭建，本质上是把这个判断力记录下来、结构化、让团队和 AI 都能用。未来设计师的竞争力，在于你的判断力有多清晰、能被多大程度地结构化。
语料库的精度，就是你的设计判断被机器理解的程度。工具在换，这件事的价值不会变。
